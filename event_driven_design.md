# Проектирование Event-Driven архитектуры для сервиса заказа такси

## Контекст

В проекте уже есть сервис такси из предыдущих лабораторных: пользователь создаёт заказ, водитель принимает его, после поездки заказ завершается. В пятой работе к API добавлялись кеширование и ограничение частоты запросов. В этой лабораторной я не стал переписывать основной сервис, потому что он уже рабочий. Вместо этого рядом добавлен небольшой сервис событий, который показывает, как такую систему можно перевести к событийному взаимодействию.

Идея простая: когда в системе происходит важное изменение, оно публикуется как событие. Остальные части системы могут подписываться на эти события и выполнять свою работу отдельно от основного API.

## Почему событийная архитектура подходит для такси

В такси почти каждое действие хорошо описывается как изменение состояния:

1. создан пользователь;
2. зарегистрирован водитель;
3. создан заказ поездки;
4. водитель принял заказ;
5. поездка завершена.

Для REST API это обычно один запрос и один ответ. Но на практике после этих действий почти всегда нужно сделать ещё что-то: обновить список активных заказов, отправить уведомление, записать событие в аудит, обновить историю поездок. Если всё это делать в одном обработчике, код становится связанным и его сложнее менять.

События позволяют отделить основное действие от дополнительных реакций. Например, создание поездки остаётся в основном сервисе, а уведомления и read-модель могут обновляться отдельно.

## Команды и события

Команда — это просьба что-то сделать. Событие — это факт, что действие уже произошло.

| Команда | Событие |
|---|---|
| CreateUser | user.created |
| RegisterDriver | driver.registered |
| CreateRide | ride.created |
| AcceptRide | ride.accepted |
| CompleteRide | ride.completed |

Например, `CreateRide` означает, что пользователь хочет создать поездку. Если сервис успешно создал запись, появляется событие `ride.created`.

## Производители событий

В полноценной версии производителем событий должен быть основной `taxi-service`. Он принимает HTTP-запросы, меняет данные в базе и после успешного изменения публикует событие в брокер.

В рамках лабораторной producer вынесен в отдельный Python-сервис:

- `event_service/producer.py` — публикует одно событие выбранного типа;
- `event_service/test_events.py` — публикует цепочку событий для проверки.

Такой вариант не ломает старую лабораторную и при этом показывает сам механизм producer → broker → consumer.

## Потребители событий

В системе выделены три потребителя:

| Потребитель | Назначение |
|---|---|
| Notification Service | мог бы отправлять уведомления пассажиру и водителю |
| Ride Read Model Service | обновляет отдельную модель для чтения поездок |
| Audit Service | получает все события и может хранить журнал действий |

В коде реализован consumer для read-модели:

- `event_service/consumer.py`.

Он слушает очередь `taxi.audit`, поэтому получает все события системы. Для read-модели он отдельно обрабатывает события поездок:

- `ride.created`;
- `ride.accepted`;
- `ride.completed`.

После обработки состояние сохраняется в файл `/data/read_model.json`.

## Брокер сообщений

В качестве брокера выбран RabbitMQ. Для этой лабораторной он удобен, потому что быстро запускается через Docker, имеет понятную модель exchange/queue/routing key и web-интерфейс для проверки.

Используется exchange:

```text
taxi.events
```

Тип exchange:

```text
topic
```

Routing key совпадает с названием события:

```text
user.created
driver.registered
ride.created
ride.accepted
ride.completed
```

## Очереди и маршрутизация

| Очередь | Routing keys | Для чего нужна |
|---|---|---|
| taxi.notifications | user.created, driver.registered, ride.created, ride.accepted, ride.completed | события для уведомлений |
| taxi.ride_read_model | ride.created, ride.accepted, ride.completed | обновление модели для чтения |
| taxi.audit | # | получение всех событий |

Для очереди аудита используется `#`, потому что аудит обычно должен видеть все события, а не только события поездок.

## Формат сообщения

Все события передаются в JSON. Внешняя структура одинаковая для всех событий:

```json
{
  "event_id": "ride.created-1710000000000",
  "event_type": "ride.created",
  "occurred_at": "2026-05-25T10:00:00+00:00",
  "source": "taxi-service",
  "payload": {
    "ride_id": 100,
    "user_id": 1,
    "from_address": "MAI",
    "to_address": "Sokol metro station",
    "status": "active"
  }
}
```

Поле `payload` зависит от конкретного события.

## Гарантии доставки

Для этой работы выбрана гарантия `at-least-once`.

Это значит, что сообщение должно быть доставлено минимум один раз. В редкой ситуации consumer может получить одно и то же событие повторно, поэтому обработчик должен быть готов к повторной обработке.

В реализации используются:

- durable exchange;
- durable queues;
- persistent messages;
- ручное подтверждение обработки через `basic_ack`;
- возврат сообщения в очередь через `basic_nack`, если обработка завершилась ошибкой.

Read-модель обновляется по `ride_id`, поэтому повторная обработка события не создаёт вторую копию поездки, а просто перезаписывает состояние.

## CQRS

CQRS здесь применим, потому что в сервисе есть разные операции для записи и чтения.

Команды, которые меняют состояние:

| Command side |
|---|
| создание пользователя |
| регистрация водителя |
| создание заказа |
| принятие заказа водителем |
| завершение поездки |

Запросы, которые только читают данные:

| Query side |
|---|
| поиск пользователя |
| получение активных заказов |
| получение истории поездок пользователя |

После выполнения команды основной сервис публикует событие. Consumer получает это событие и обновляет read-модель. В лабораторной read-модель хранится в JSON-файле, чтобы её было легко посмотреть командой `cat`. В реальном проекте это могла бы быть отдельная таблица, MongoDB-коллекция или Redis.

## Поток событий для поездки

Типовой сценарий выглядит так:

1. Пассажир создаёт заказ.
2. Публикуется `ride.created`.
3. Consumer добавляет поездку в `active_rides`.
4. Водитель принимает заказ.
5. Публикуется `ride.accepted`.
6. Consumer переносит поездку в `accepted_rides`.
7. Поездка завершается.
8. Публикуется `ride.completed`.
9. Consumer переносит поездку в `completed_rides`.

Так формируется отдельное представление для чтения текущего состояния поездок.

## Что реализовано

В проект добавлено:

- RabbitMQ в `docker-compose.yaml`;
- Python producer для публикации событий;
- Python consumer для обработки событий поездок;
- тестовый сценарий публикации событий;
- read-модель, которая обновляется через события.

Реализация небольшая, но она показывает основные элементы Event-Driven подхода: producer, broker, exchange, queue, routing key, consumer и обновление read-модели.
