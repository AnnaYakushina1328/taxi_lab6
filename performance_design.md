# Лабораторная работа 5. Производительность сервиса такси

Вариант 16: система заказа такси.

## 1. Что оптимизируется

В этой работе я рассматриваю не всю систему целиком, а те места API, которые чаще всего вызываются и сильнее всего зависят от базы данных.

В сервисе есть три основные сущности:

- пользователь;
- водитель;
- поездка.

Для пятой лабораторной важнее всего операции чтения. Именно они могут часто повторяться и поэтому хорошо подходят для кеширования.

## 2. Часто вызываемые операции

Я выделил такие hot paths:

| Endpoint | Почему он важен |
|---|---|
| `GET /users?login=...` | поиск пользователя по логину может использоваться при проверке аккаунта |
| `GET /users?name_mask=...` | поиск по имени может вызываться администратором или диспетчером |
| `GET /rides?status=active` | водители часто запрашивают список доступных заказов |
| `GET /rides?user_id=...` | пользователь может открывать историю своих поездок |

Самый чувствительный endpoint здесь — `GET /rides?status=active`. В реальном такси такой список быстро меняется, но при этом может запрашиваться очень часто.

## 3. Где могут быть задержки

Медленными могут быть не сами обработчики, а операции, которые они вызывают:

- запросы к PostgreSQL;
- поиск пользователей по текстовым полям;
- чтение активных поездок;
- чтение истории поездок;
- изменение статуса поездки.

Создание, принятие и завершение поездки тоже важны, потому что после них старые данные в кеше уже нельзя считать актуальными.

## 4. Целевые требования

Для учебного проекта я задал такие ориентиры:

| Метрика | Ориентир |
|---|---|
| Ответ при попадании в кеш | до 20 мс |
| Ответ при промахе кеша | до 100 мс |
| Операции записи | до 200 мс |
| Лимит для активных поездок | 100 запросов в минуту |
| Поведение при превышении лимита | HTTP 429 |

Значения выбраны как простые ориентиры для проверки, а не как production SLA.

## 5. Что кешируется

Кешируются только GET-запросы. Запросы, которые меняют данные, не кешируются.

| Данные | Ключ кеша | TTL |
|---|---|---|
| Пользователь по логину | `users:login:{login}` | 60 секунд |
| Пользователи по маске имени | `users:name_mask:{mask}` | 60 секунд |
| Активные поездки | `rides:active` | 15 секунд |
| История поездок пользователя | `rides:history:{user_id}` | 60 секунд |

Для активных поездок TTL меньше, потому что список заказов может быстро измениться. Для пользователей и истории поездок можно держать кеш дольше.

## 6. Выбранная стратегия кеширования

Используется Cache-Aside.

Работает это так:

1. Приходит GET-запрос.
2. Сервис строит ключ кеша.
3. Если по этому ключу есть актуальный ответ, сервис возвращает его сразу.
4. Если ответа нет, сервис делает запрос в PostgreSQL.
5. Полученный JSON сохраняется в кеш.
6. Клиент получает ответ.

Эта стратегия выбрана из-за простоты. Для лабораторной работы не требуется отдельный Redis, поэтому кеш хранится в памяти процесса.

## 7. Инвалидация

Кеш очищается не по отдельному ключу, а по префиксу. Это проще и безопаснее для учебного проекта.

| Операция | Что очищается | Почему |
|---|---|---|
| `POST /users` | `users:` | появился новый пользователь |
| `POST /drivers` | `drivers:` | появился новый водитель |
| `POST /rides` | `rides:` | появился новый заказ |
| `PATCH /rides/{id}/accept` | `rides:` | заказ больше не должен считаться свободным |
| `PATCH /rides/{id}/complete` | `rides:` | изменилась история и статус поездки |

Такой подход немного грубее точечной инвалидации, но для небольшого сервиса он понятнее и меньше рискует оставить устаревшие данные.

## 8. Реализация кеша

Кеш находится в файле:

- `src/performance/simple_performance.hpp`

Что есть в реализации:

- хранение готового JSON-ответа;
- TTL для каждой записи;
- удаление устаревшей записи при чтении;
- очистка по префиксу;
- защита через `std::mutex`;
- счётчики hits и misses.

Для проверки в HTTP-ответ добавляется заголовок:

- `X-Cache: MISS`, если был запрос к базе;
- `X-Cache: HIT`, если ответ взят из кеша.

## 9. Rate limiting

Rate limiting добавлен для endpoint-а:

- `GET /rides?status=active`

Именно его логично ограничивать, потому что список активных заказов может часто обновляться на стороне водителя.

Алгоритм:

- Fixed Window Counter.

Лимит:

- 100 запросов в минуту.

Если лимит превышен, сервис возвращает:

- `429 Too Many Requests`.

В ответ добавляются заголовки:

- `X-RateLimit-Limit`;
- `X-RateLimit-Remaining`;
- `X-RateLimit-Reset`;
- `Retry-After`.

## 10. Почему Fixed Window Counter

Для этой лабораторной я выбрал Fixed Window Counter, потому что он простой и легко проверяется через curl.

Минусы у него тоже есть. Например, на границе двух минут можно отправить больше запросов, чем хотелось бы. Для реальной системы лучше подошёл бы sliding window или token bucket. Но для учебного проекта fixed window достаточно показывает сам принцип ограничения нагрузки.

## 11. Влияние на производительность

Кеширование помогает уменьшить количество повторных запросов к PostgreSQL. Если несколько клиентов подряд запрашивают активные поездки, сервис не каждый раз идёт в базу, а может вернуть готовый JSON из памяти.

Rate limiting решает другую задачу. Он не ускоряет один запрос, но защищает сервис от слишком частых вызовов одного endpoint-а. Если клиент делает больше 100 запросов в минуту, сервис начинает отвечать `429`, а не продолжает нагружать обработчик и базу.

## 12. Метрики

Для оценки результата можно смотреть такие метрики:

| Метрика | Что показывает |
|---|---|
| cache_hits | сколько раз ответ взяли из кеша |
| cache_misses | сколько раз пришлось идти в базу |
| cache_hit_rate | доля ответов из кеша |
| request_latency_ms | время ответа API |
| db_query_count | количество обращений к PostgreSQL |
| http_429_total | сколько запросов ограничил rate limiter |
| rate_limit_remaining | сколько запросов осталось в текущем окне |

Hit rate считается так:

`cache_hits / (cache_hits + cache_misses)`

Например, если было 80 попаданий и 20 промахов, hit rate равен 80%.

## 13. Ограничения решения

Текущая реализация подходит для лабораторной работы, но у неё есть ограничения:

- кеш хранится только в памяти одного процесса;
- после перезапуска сервиса кеш очищается;
- если запустить несколько экземпляров сервиса, у каждого будет свой кеш;
- rate limiter тоже хранит счётчики только внутри одного процесса.

Для production-версии я бы вынес кеш и счётчики rate limiting в Redis. Тогда несколько экземпляров сервиса могли бы использовать общие данные.
