# Лабораторная работа 5. Производительность сервиса такси

Вариант 16: система заказа такси.

## Что оптимизировалось

В этой лабораторной работе я добавила к сервису такси кеширование и ограничение частоты запросов.

Сервис работает с тремя основными сущностями:

- пользователь;
- водитель;
- поездка.

В первую очередь оптимизировались GET-запросы, потому что они чаще повторяются и могут создавать лишнюю нагрузку на PostgreSQL.

## Частые операции

Я выделила несколько операций, которые логично считать hot paths:

| Endpoint | Почему выбран |
|---|---|
| `GET /users?login=...` | поиск пользователя по логину может часто использоваться при проверке аккаунта |
| `GET /users?name_mask=...` | поиск пользователей по части имени |
| `GET /rides?status=active` | водители часто запрашивают список доступных заказов |
| `GET /rides?user_id=...` | пассажир может смотреть историю своих поездок |

Самый чувствительный endpoint — `GET /rides?status=active`, потому что список активных заказов может часто открываться водителями.

## Где возможны задержки

Основные задержки могут появляться не в самом обработчике, а при обращении к базе данных:

- поиск пользователя по логину;
- поиск пользователей по имени;
- получение активных поездок;
- получение истории поездок пользователя;
- изменение статуса поездки.

Операции создания, принятия и завершения поездки важны ещё и потому, что после них старый кеш уже нельзя считать актуальным.

## Кешируемые данные

Кешируются только запросы чтения:

| Данные | Ключ кеша | TTL |
|---|---|---|
| Пользователь по логину | `users:login:{login}` | 60 секунд |
| Пользователи по маске имени | `users:name_mask:{mask}` | 60 секунд |
| Активные поездки | `rides:active` | 15 секунд |
| История поездок пользователя | `rides:history:{user_id}` | 60 секунд |

Для активных поездок TTL сделан коротким, потому что список доступных заказов быстро меняется. Для пользователей и истории поездок можно использовать TTL побольше.

## Стратегия кеширования

Используется Cache-Aside.

Логика простая:

1. Приходит GET-запрос.
2. Сервис строит ключ кеша.
3. Если по ключу есть актуальный ответ, сервис возвращает его сразу.
4. Если ответа нет, сервис обращается к PostgreSQL.
5. Полученный JSON сохраняется в кеш.
6. Клиент получает ответ.

Для этой лабораторной кеш хранится в памяти процесса. Redis не добавлялся специально, чтобы не усложнять запуск проекта.

## Инвалидация кеша

Кеш очищается после операций, которые меняют данные:

| Операция | Что очищается |
|---|---|
| `POST /users` | `users:` |
| `POST /drivers` | `drivers:` |
| `POST /rides` | `rides:` |
| `PATCH /rides/{id}/accept` | `rides:` |
| `PATCH /rides/{id}/complete` | `rides:` |

Я использовала очистку по префиксу. Это не самый точечный вариант, зато он простой и безопасный для учебного проекта: меньше риск оставить старые данные в кеше.

## Реализация кеша

Кеш реализован в файле:

- `src/performance/simple_performance.hpp`

В реализации есть:

- хранение готового JSON-ответа;
- TTL для каждой записи;
- удаление устаревшей записи при чтении;
- очистка по префиксу;
- защита через `std::mutex`;
- счётчики cache hits и cache misses.

Для проверки в HTTP-ответ добавляется заголовок:

- `X-Cache: MISS` — ответ собран через запрос к базе;
- `X-Cache: HIT` — ответ взят из кеша.

## Rate limiting

Rate limiting добавлен для endpoint-а:

- `GET /rides?status=active`

Этот endpoint может часто вызываться водителями, поэтому его логично ограничить.

Алгоритм:

- Fixed Window Counter.

Лимит:

- 100 запросов в минуту.

При превышении лимита сервис возвращает:

- `429 Too Many Requests`.

Также добавляются заголовки:

- `X-RateLimit-Limit`;
- `X-RateLimit-Remaining`;
- `X-RateLimit-Reset`;
- `Retry-After`.

## Почему выбран Fixed Window Counter

Для лабораторной работы этот алгоритм подходит, потому что он простой и легко проверяется через `curl`.

Недостаток у него тоже есть: на границе двух минут клиент теоретически может отправить больше запросов, чем задано лимитом. Для production-версии лучше использовать sliding window или token bucket, но для учебной реализации fixed window достаточно показывает принцип ограничения нагрузки.

## Влияние на производительность

Кеширование уменьшает количество повторных запросов к PostgreSQL. Например, если несколько раз подряд запросить активные поездки, второй запрос уже может быть отдан из памяти.

Rate limiting решает другую задачу: он не ускоряет отдельный запрос, но защищает сервис от слишком частых обращений к одному endpoint-у. После превышения лимита сервис не продолжает нагружать БД, а возвращает `429`.

## Метрики

Для оценки результата можно смотреть:

| Метрика | Что показывает |
|---|---|
| `cache_hits` | сколько раз ответ взяли из кеша |
| `cache_misses` | сколько раз пришлось идти в базу |
| `cache_hit_rate` | доля ответов из кеша |
| `request_latency_ms` | время ответа API |
| `db_query_count` | количество обращений к PostgreSQL |
| `http_429_total` | количество запросов, ограниченных rate limiter |
| `rate_limit_remaining` | сколько запросов осталось в текущем окне |

Hit rate считается так:

`cache_hits / (cache_hits + cache_misses)`

Например, если было 80 попаданий и 20 промахов, hit rate равен 80%.

## Ограничения решения

Текущая реализация подходит для лабораторной работы, но у неё есть ограничения:

- кеш хранится только в памяти одного процесса;
- после перезапуска сервиса кеш очищается;
- если запустить несколько экземпляров сервиса, у каждого будет свой кеш;
- rate limiter тоже хранит счётчики только внутри одного процесса.

В реальном проекте я бы вынесла кеш и счётчики rate limiting в Redis.
