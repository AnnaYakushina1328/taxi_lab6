## Вариант 16 — Система заказа такси

В рамках варианта 16 система заказа такси должна работать с тремя основными сущностями:
- пользователь;
- водитель;
- поездка.

Поддерживаемые операции:
- создание нового пользователя;
- поиск пользователя по логину;
- поиск пользователя по маске имени и фамилии;
- регистрация водителя;
- создание заказа поездки;
- получение активных заказов;
- принятие заказа водителем;
- получение истории поездок пользователя;
- завершение поездки.

## Цель проектирования документной модели

Цель проектирования — адаптировать сервис заказа такси под MongoDB, сохранив ранее реализованную логику API на userver и не удаляя PostgreSQL из проекта. MongoDB в данной лабораторной работе используется как дополнительная база данных в составе одного сервиса.

Документная модель должна:
- поддерживать основные операции варианта;
- быть понятной и компактной;
- не приводить к избыточному дублированию крупных сущностей;
- позволять удобно выполнять CRUD-операции;
- демонстрировать осознанный выбор между embedded documents и references.

## Выбранная модель MongoDB

Для сервиса такси выбраны три основные коллекции:
- `users`
- `drivers`
- `rides`

Такой подход соответствует предметной области и уже существующей структуре проекта из прошлой лабораторной работы, где логика также построена вокруг сущностей User, Driver и Ride.

## Коллекция `users`

Коллекция `users` хранит пользователей сервиса такси.

### Пример структуры документа

```json
{
  "_id": { "$oid": "671111111111111111111111" },
  "login": "ivan.petrov",
  "password": "hashed_password_value",
  "first_name": "Иван",
  "last_name": "Петров",
  "created_at": { "$date": "2026-04-18T10:00:00Z" },
  "status": "active",
  "profile": {
    "email": "ivan.petrov@example.com",
    "phone": "+79990000001"
  }
}
```

### Назначение полей

- `_id` — уникальный идентификатор пользователя;
- `login` — уникальный логин пользователя;
- `password` — пароль или его хэш;
- `first_name` — имя;
- `last_name` — фамилия;
- `created_at` — дата создания записи;
- `status` — состояние пользователя;
- `profile` — встроенный объект с дополнительной информацией о пользователе.

### Почему `profile` сделан embedded document

Объект `profile` хранится внутри документа пользователя как embedded document, потому что:
- он принадлежит только одному пользователю;
- имеет маленький размер;
- всегда используется вместе с пользователем;
- не требует отдельного жизненного цикла и отдельных запросов.

Для такой связи embedded document подходит лучше, чем отдельная коллекция.

## Коллекция `drivers`

Коллекция `drivers` хранит данные водителей.

### Пример структуры документа

```json
{
  "_id": { "$oid": "672222222222222222222222" },
  "login": "driver.alex",
  "first_name": "Алексей",
  "last_name": "Сидоров",
  "status": "available",
  "registered_at": { "$date": "2026-04-18T10:15:00Z" },
  "car": {
    "brand": "Kia",
    "model": "Rio",
    "plate_number": "A123BC777",
    "color": "white"
  },
  "profile": {
    "phone": "+79990000021",
    "license_number": "77AB123456"
  }
}
```

### Назначение полей

- `_id` — уникальный идентификатор водителя;
- `login` — логин водителя;
- `first_name` — имя;
- `last_name` — фамилия;
- `status` — состояние водителя;
- `registered_at` — дата регистрации водителя;
- `car` — встроенный объект с данными автомобиля;
- `profile` — встроенный объект с дополнительными сведениями о водителе.

### Почему `car` и `profile` сделаны embedded

Объекты `car` и `profile` встроены в документ водителя, потому что:
- они тесно связаны именно с конкретным водителем;
- имеют небольшой размер;
- чаще всего читаются вместе с водителем;
- не используются как самостоятельные сущности вне контекста водителя.

Поэтому в данном случае отдельные коллекции для автомобиля и профиля только усложнили бы модель без явной пользы.

## Коллекция `rides`

Коллекция `rides` хранит информацию о поездках.

### Пример структуры документа

```json
{
  "_id": { "$oid": "673333333333333333333333" },
  "user_id": { "$oid": "671111111111111111111111" },
  "driver_id": { "$oid": "672222222222222222222222" },
  "status": "accepted",
  "created_at": { "$date": "2026-04-18T11:00:00Z" },
  "accepted_at": { "$date": "2026-04-18T11:03:00Z" },
  "completed_at": null,
  "route": {
    "pickup_address": "Москва, Ленинградский проспект, 1",
    "dropoff_address": "Москва, Тверская, 10"
  },
  "fare": {
    "amount": 650,
    "currency": "RUB"
  },
  "user_snapshot": {
    "login": "ivan.petrov",
    "first_name": "Иван",
    "last_name": "Петров"
  },
  "driver_snapshot": {
    "login": "driver.alex",
    "first_name": "Алексей",
    "last_name": "Сидоров",
    "car_brand": "Kia",
    "car_model": "Rio",
    "plate_number": "A123BC777"
  },
  "events": [
    {
      "type": "created",
      "at": { "$date": "2026-04-18T11:00:00Z" }
    },
    {
      "type": "accepted",
      "at": { "$date": "2026-04-18T11:03:00Z" }
    }
  ]
}
```

### Назначение полей

- `_id` — идентификатор поездки;
- `user_id` — ссылка на пользователя;
- `driver_id` — ссылка на водителя;
- `status` — статус поездки;
- `created_at` — дата создания заказа;
- `accepted_at` — дата принятия поездки водителем;
- `completed_at` — дата завершения поездки;
- `route` — встроенный объект с адресами поездки;
- `fare` — встроенный объект со стоимостью;
- `user_snapshot` — встроенная копия части данных пользователя;
- `driver_snapshot` — встроенная копия части данных водителя;
- `events` — массив событий по поездке.

### Почему `user_id` и `driver_id` сделаны references

Связи `ride -> user` и `ride -> driver` реализованы через references, потому что:

#### 1. Поездка — самостоятельная сущность

Поездка имеет собственный жизненный цикл:
- создаётся при оформлении заказа;
- изменяется при принятии водителем;
- изменяется при завершении;
- участвует в отдельных запросах и истории поездок.

#### 2. У пользователя может быть много поездок

Если хранить все поездки внутри документа пользователя, документ будет постоянно расти. Для истории поездок это неудобно и плохо масштабируется.

#### 3. У водителя также может быть много поездок

Один водитель выполняет множество поездок. Если хранить все поездки внутри документа водителя, документ водителя тоже станет слишком большим.

#### 4. Пользователи и водители должны существовать независимо

Пользователя и водителя нужно хранить отдельно от поездки:
- пользователь ищется по логину;
- пользователь ищется по маске имени и фамилии;
- водитель регистрируется отдельно;
- активные поездки и история поездок должны собираться как отдельные сущности.

Поэтому хранить `users`, `drivers` и `rides` как отдельные коллекции правильнее, чем полностью вкладывать их друг в друга.

### Почему в `rides` дополнительно оставлены embedded snapshots

Несмотря на references, в `rides` дополнительно используются:
- `user_snapshot`
- `driver_snapshot`

Это сделано как компромисс между нормализацией и удобством чтения.

### Причины

- при получении истории поездок пользователя часто сразу нужны имя пользователя, водитель и автомобиль;
- snapshots позволяют отдать часть данных без дополнительного `$lookup`;
- snapshot фиксирует состояние данных на момент поездки.

### Почему это допустимо

Дублирование здесь небольшое и контролируемое:
- в snapshot хранятся только короткие поля;
- основными источниками истины всё равно остаются коллекции `users` и `drivers`;
- references сохраняются, поэтому связь между сущностями не теряется.

## Сводка выбора embedded/references

### Embedded documents используются для:

- `users.profile`
- `drivers.car`
- `drivers.profile`
- `rides.route`
- `rides.fare`
- `rides.user_snapshot`
- `rides.driver_snapshot`
- `rides.events`

Причина: маленький размер, тесная связь с родительским документом, чтение вместе с основной сущностью.

### References используются для:

- `rides.user_id -> users._id`
- `rides.driver_id -> drivers._id`

Причина: независимый жизненный цикл сущностей, большое количество связей, необходимость отдельных запросов и отсутствие бесконтрольного роста документов.

## Почему не выбрана модель «все поездки внутри users»

Такой вариант был рассмотрен, но не выбран, потому что:
- история поездок пользователя может быть большой;
- поездка отдельно меняет статус;
- активные заказы и завершённые поездки удобнее хранить отдельно;
- усложняется работа с водителями и назначением поездки.

## Почему не выбрана модель «все поездки внутри drivers»

Такой вариант тоже не выбран, потому что:
- один водитель может принять большое количество поездок;
- документ водителя начинает хранить длинную историю;
- неудобно получать историю поездок конкретного пользователя;
- операции принятия и завершения поездки становятся сложнее.

## Итоговое решение

Для MongoDB в сервисе заказа такси выбрана гибридная модель:
- основные сущности `users`, `drivers` и `rides` вынесены в отдельные коллекции;
- для связей между поездками, пользователями и водителями используются references;
- для маленьких и часто читаемых частей данных используются embedded documents.

Такое решение:
- хорошо соответствует варианту 16;
- поддерживает все основные операции задания;
- удобно для реализации на userver;
- позволяет показать в лабораторной работе осознанный выбор между embedded documents и references.
